# 2026-06-01 k6 / wrk 全链路压测报告

## 结论摘要

本轮压测覆盖 Spring Boot 主后端的健康检查、驾驶员读链路、普通管理员读链路、超级管理员日志/RAG 读链路、登录基准、AI actions、RAG 检索和 AI stream 模型生成链路。

整体结论：

- k6 全 API 混合压测通过：`http_req_failed_rate=0.000`，`checks_rate=1.000`，p95 为 `50.967ms`。
- AI actions 尾延迟已明显下降：wrk 平均延迟 `13.88ms`，无 timeout，吞吐 `656.55 req/s`。
- AI/RAG 分段压测通过严格阈值：AI actions p95 `12.7ms`，RAG 检索 p95 `1197.8ms`，AI stream p95 `5166.7ms`。
- 管理员/超级管理员混合读仍出现 wrk timeout 候选，但没有非 2xx 响应；当前更像高并发下的尾延迟/连接排队，而不是权限或业务错误。
- 登录 wrk 基准触发大量非 2xx，这是预期行为：登录限流与失败惩罚会在极高并发下保护 BCrypt，不应作为业务可用性失败理解。

## 暴露问题

1. RAG 检索依赖后端启动参数

   压测前必须显式开启 `RAG_RETRIEVAL_ENABLED=true`，否则 RAG 数据集能写入但查询为空。已在压测编排脚本中补充失败提示。

2. RAG 检索仍有尾延迟

   k6 p95 为 `1197.8ms`，p99 为 `1951.1ms`。当前还没到不可用，但对交互式 AI 场景已经偏高。

3. 管理员/超级管理员 wrk timeout 不是 5xx

   endpoint 级统计显示非 2xx 为 0，timeout 候选相对均匀。下一步应结合后端慢查询日志、Hikari 指标、Tomcat 线程池指标和 ES 查询耗时拆分来确认是 DB、ES 还是连接排队。

4. wrk timeout 归因仍是候选归因

   wrk Lua 的 `response` 回调不会在 timeout 请求上触发，因此报告中的 missing 只能定位候选端点，不能替代服务端 access log 或 Micrometer endpoint 标签。

## 后续建议

- RAG：记录每次查询的 `embedding_ms`、`es_search_ms`、`rerank_ms` 和 `total_ms`，避免只看到总耗时。
- RAG：调小候选集倍数或对 rerank 做上限保护，给交互式 AI 设更稳的 p95。
- 超级管理员读接口：日志 overview、recent logs、RAG document list 使用只读投影或短 TTL 缓存。
- 管理员读接口：继续观察 Hikari active/pending、Tomcat busy threads、数据库慢查询，确认 wrk timeout 是客户端压力侧还是服务端资源侧。
- 登录压测：保留独立场景，不纳入常规业务成功率；重点观察限流命中率和 BCrypt 执行次数。
