# 2026-05-30 k6 / wrk 压测报告

## 测试范围

本轮压测覆盖 Spring Boot 主后端的 HTTP API，目标地址为 `http://127.0.0.1:8080`。测试以本地开发环境为基准，不代表生产容量。

覆盖链路：

| 类别 | 覆盖内容 |
| --- | --- |
| 健康检查 | `/actuator/health/liveness`、`/actuator/health/readiness` |
| 认证链路 | 注册测试驾驶员账号、驾驶员/管理员/超级管理员登录、登录基线 |
| 驾驶员端 | 当前用户信息、驾驶员档案、车辆绑定、违法、罚款、缴费、扣分、申诉列表 |
| 普通管理员端 | 用户、驾驶员、车辆、违法、罚款、缴费、扣分、申诉、违法类型、权限、角色、系统配置读接口 |
| 超级管理员端 | 系统日志概览、登录日志、操作日志、RAG 概览、RAG 文档列表 |

AI actions 接口在当前压测环境中未纳入阈值：本轮后端以 `AI_PROVIDER=noop` 启动，手工探测 `/api/ai/chat/actions?message=...&webSearch=false` 返回 `500`。AI/LLM 链路应在模型服务可用后单独压测，避免把模型不可用误判为 HTTP 层性能问题。

## wrk 结果

### 登录 POST 基准

参数：`4 threads`、`32 connections`、`30s`。

| 指标 | 结果 |
| --- | ---: |
| 总请求数 | `517` |
| 吞吐 | `15.97 req/s` |
| 平均延迟 | `1.83 s` |
| 最大延迟 | `2.00 s` |
| 读吞吐 | `21.80 KB/s` |
| Socket timeout | `32` |
| 非 2xx/3xx | `1` |

登录结论：登录接口吞吐明显低于读接口，主要原因是登录会执行 `BCrypt` 密码校验，当前成本参数为 `12`。这对安全是合理的，但需要通过限流、登录失败惩罚、登录接口隔离线程池或网关保护避免被高并发登录拖垮业务读接口。

### 管理员读接口混合基准

参数：`4 threads`、`64 connections`、`30s`。

| 指标 | 结果 |
| --- | ---: |
| 总请求数 | `30,561` |
| 吞吐 | `949.81 req/s` |
| 平均延迟 | `107.53 ms` |
| 最大延迟 | `2.00 s` |
| 读吞吐 | `2.06 MB/s` |
| Socket timeout | `11` |
| 非 2xx/3xx | `24` |

读链路结论：管理员读接口在 64 连接下吞吐接近 `950 req/s`，平均延迟约 `108 ms`。存在少量超时和非 2xx/3xx 响应，说明本地开发环境在突发并发下已经出现尾延迟，后续应重点检查数据库连接池、慢查询、分页查询是否缺索引，以及全量列表接口是否需要限制页大小或加缓存。

## 结论

1. 读接口整体表现可接受：k6 混合链路 P95 为 `36.01 ms`，P99 为 `117.65 ms`。
2. 登录接口是明显瓶颈：wrk 下 `32` 并发仅 `15.97 req/s`，符合 BCrypt 高成本校验的预期。
3. 本地开发环境已出现少量尾延迟和瞬时失败：wrk 管理员读混合有 `11` 次 timeout、`24` 次非 2xx/3xx；k6 也记录到 `10` 次业务读失败。
4. AI actions 未纳入本轮阈值：当前 noop AI 环境返回 `500`，建议在 Ollama/实际模型服务可用后单独压测 AI/RAG 链路。

## 建议

| 优先级 | 建议 |
| --- | --- |
| P0 | 登录接口加限流和失败惩罚，避免 BCrypt 被恶意并发放大 |
| P1 | 对驾驶员/管理员高频读接口增加慢查询日志与连接池指标 |
| P1 | 为分页读接口统一限制 `size` 上限，避免全量返回造成尾延迟 |
| P2 | 对日志、RAG 文档列表等超级管理员读接口增加缓存或只读投影 |
| P2 | AI/RAG 单独设计压测场景，区分 HTTP 编排耗时、检索耗时、模型生成耗时 |
