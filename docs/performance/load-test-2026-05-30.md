# 2026-05-30 k6 / wrk 全链路压测报告

## 最新维护记录

2026-05-31 对压测链路做了以下修正：

- `/api/ai/chat/actions` 不再直接硬依赖 Ollama 同步调用，改为走统一 AI Provider/fallback 链路；provider 失败时返回空 actions，避免接口 500。
- `scripts/performance/run-load-tests.ps1` 在 AI/RAG 阶段默认设置 `PERF_STRICT=true`。
- AI/RAG 压测前会调用 `scripts/performance/seed-rag-load-dataset.ps1`，写入真实 RAG 检索资料，覆盖“有文档、有 chunk、有向量、有 rerank”的检索路径。
- `scripts/k6/ai-rag-staged-load.js` 将 AI stream 拆分统计为真实 `ollama` 调用成功率和 `noop fallback` 比例。
- `scripts/wrk/read-mix.lua` 与 `scripts/wrk/super-read-mix.lua` 在结束时输出 `Non-2xx responses by endpoint`，用于定位管理员和超级管理员混合读中的失败端点。

## 测试范围

本报告覆盖 Spring Boot 主后端的本地 HTTP API，目标地址为 `http://127.0.0.1:8080`。测试环境是 Windows 本地开发机、本地 MySQL、Docker Desktop 中的 Elasticsearch/Redpanda/Debezium 等组件，不代表生产容量。

| 类别 | 覆盖内容 |
| --- | --- |
| 健康检查 | `/actuator/health/liveness`、`/actuator/health/readiness` |
| 认证链路 | 驾驶员、普通管理员、超级管理员登录；登录接口独立 wrk 基准 |
| 驾驶员端 | 当前用户、驾驶员档案、车辆绑定、违法、罚款、缴费、扣分、申诉列表 |
| 普通管理员端 | 用户、驾驶员、车辆、违法、罚款、缴费、扣分、申诉、违法类型、权限、角色、系统配置读接口 |
| 超级管理员端 | 系统日志概览、登录日志、操作日志、RAG 概览、RAG 文档列表 |
| RAG / AI | RAG query、AI actions、AI stream 三段分离压测 |

## AI/RAG 复测口径

新的 AI/RAG 复测必须同时观察这些指标：

| 分段 | 观察点 |
| --- | --- |
| AI HTTP 编排 | `/api/ai/chat/actions` 必须在 `PERF_STRICT=true` 下保持 2xx 且 `success=true` |
| RAG 检索 | `/api/rag/query` 必须命中 `rag-real-dataset.json` 写入的真实资料 |
| RAG 数据准备 | 管理端写入文档后产生 chunk，并由 embedding 批处理写入向量索引 |
| rerank | 查询结果经过当前 `RerankProvider`，避免只测空结果或纯 HTTP 包装 |
| AI stream | 报告中分别查看 `ai_model_ollama_ok` 与 `ai_model_noop_fallback_ok` |
