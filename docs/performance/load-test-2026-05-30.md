# 2026-05-30 k6 / wrk 全链路压测报告

## 测试范围

本轮压测覆盖 Spring Boot 主后端的本地 HTTP API，目标地址为 `http://127.0.0.1:8080`。测试环境是 Windows 本地开发机和本地 MySQL，不代表生产容量。

覆盖链路：

| 类别 | 覆盖内容 |
| --- | --- |
| 健康检查 | `/actuator/health/liveness`、`/actuator/health/readiness` |
| 认证链路 | 驾驶员、普通管理员、超级管理员登录；登录接口独立 wrk 基准 |
| 驾驶员端 | 当前用户、驾驶员档案、车辆绑定、违法、罚款、缴费、扣分、申诉列表 |
| 普通管理员端 | 用户、驾驶员、车辆、违法、罚款、缴费、扣分、申诉、违法类型、权限、角色、系统配置读接口 |
| 超级管理员端 | 系统日志概览、登录日志、操作日志、RAG 概览、RAG 文档列表 |

## k6 AI/RAG 分段结果

参数：AI actions `1 req/s`，RAG query `1 req/s`，AI stream `1 req/s`，持续 `20s`。

| 分段 | 成功率 | 平均耗时 | P95 | P99 | 说明 |
| --- | ---: | ---: | ---: | ---: | --- |
| AI HTTP 编排 `/api/ai/chat/actions` | `0%` | `49.50 ms` | `53.25 ms` | `54.46 ms` | 当前接口全部返回非 2xx，需要作为功能缺陷处理 |
| RAG 检索 `/api/rag/query` | `100%` | `2.70 ms` | `3.62 ms` | `4.35 ms` | 当前命中空结果也能稳定返回 |
| 模型流式 `/api/ai/chat/stream` | `100%` | `6.83 ms` | `8.85 ms` | `12.24 ms` | 当前返回很快，实际更像后端降级/短响应，不代表真实 Ollama 生成吞吐 |

AI/RAG 结论：RAG 查询链路稳定，stream 接口可用；`/api/ai/chat/actions` 当前是明确失败点。后续要把 AI actions 的 500 修复后，再开启 `PERF_STRICT=true` 让该脚本进入严格阈值模式。

## 主要结论

1. 驾驶员、管理员、超级管理员读链路整体可承受本地中等并发，P95 延迟保持在较低水平。
2. 登录接口是独立瓶颈，当前 16 并发下约 `17.10 req/s`，符合安全型登录设计预期。
3. RAG query 当前非常快，但结果为空时的吞吐不能代表大规模向量检索、rerank 和真实文档命中的成本。
4. AI stream 链路当前可用，但响应过快，需确认是否实际调用 Ollama；这轮不能作为真实模型吞吐结论。
5. AI actions 是当前明确功能缺陷：k6 成功率 `0%`，wrk 非 2xx/3xx 为 `586/586`。
6. 管理员和超级管理员 wrk 读混合存在少量非 2xx/3xx，需要结合后端日志定位具体端点。
