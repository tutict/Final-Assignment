# 2026-05-31 k6 / wrk 全链路压测报告

## 测试结论

本轮在 Windows 本地开发机执行，后端目标为 `http://127.0.0.1:8080`，MySQL 使用本机服务，Redis 与 Elasticsearch 通过 Docker Desktop 启动，wrk 通过 Docker 容器运行。

核心读链路、RAG 真实检索链路、AI stream 真实 Ollama 链路均可用；`PERF_STRICT=true` 下暴露出一个明确瓶颈：`/api/ai/chat/actions` 在真实模型参与后 p95 达到 `3171.5 ms`，超过脚本阈值 `p95 < 1200 ms`，因此 `ai-rag-staged-load` 以 k6 exit code `99` 标记为阈值失败。

## 命令

```powershell
docker compose -f scripts\dev-compose.yml up -d redis elasticsearch --wait --wait-timeout 180

powershell -ExecutionPolicy Bypass -File scripts\performance\run-load-tests.ps1 `
  -Duration 20s `
  -DriverVus 8 `
  -AdminVus 6 `
  -SuperVus 2 `
  -LoginRate 0 `
  -IncludeModel
```

`run-load-tests.ps1` 会先写入 `scripts/performance/rag-real-dataset.json`，再验证 `/api/rag/query` 至少命中 1 条结果。AI/RAG 阶段固定启用 `PERF_STRICT=true`，并分别统计 HTTP 编排、RAG 检索、模型生成、真实 Ollama 调用和 noop fallback。

## 问题与下一步

1. `/api/ai/chat/actions` 当前在真实模型参与时尾延迟明显，wrk 8 连接下出现大量 timeout。建议把页面跳转、填单等动作识别优先走规则/意图模板，只有低置信度时再调用模型。
2. 管理员读混合出现 19 个 timeout，但 endpoint 非 2xx 为 `none`，说明不是权限或路由失败，更像本地资源竞争或数据库/ES 尾延迟。
3. 登录压测已经验证限流和失败惩罚会大量拒绝高频登录请求；后续应单独补充“合法低速登录成功率”和“恶意高速登录拒绝率”两个报告项，避免把保护性拒绝误判为业务失败。
